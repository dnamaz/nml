# NML Implementation Roadmap — Phases 1–5

## Overview

This document describes the 5-phase capability expansion completed in March 2026.
Each phase was implemented, validated, and tested independently before the next phase began.
Phases build on each other: Phase 2 GPU kernels operate on the 4D NCHW tensors introduced in Phase 1; Phase 4 ONNX import benefits from Phase 1 broadcasting; Phase 5 embeds the runtime that includes all prior phases.

The result is a complete runtime at v1.0.0: 89 opcodes, GPU acceleration on three backends (CUDA, SYCL, Metal), ONNX model import, a clean C library API, and a Python binding.

---

## Phase 1 — Foundation

**Goal:** Close the gaps between NML's op set and what real ML models require for deployment.

### Broadcasting (MADD, MSUB, EMUL, EDIV)

Prior behaviour: both operands must have identical shapes.
New behaviour: NumPy-style right-alignment. Dimensions are compared right-to-left; a dimension of 1 broadcasts to match the other. This enables bias addition (`[batch, features]` MADD `[features]`) without explicit reshape.

Examples:
- `[4,3]` MADD `[3]` → broadcasts to `[4,3]`
- `[1,4,3]` EMUL `[4,3]` → broadcasts to `[1,4,3]`
- `[2,1,3]` EMUL `[2,4,3]` → broadcasts to `[2,4,3]`

### 4D NCHW Vision Ops (CONV, POOL, UPSC, PADZ)

Prior behaviour: 2D inputs only.
New behaviour: All four vision ops accept 4D tensors in standard NCHW layout:
- Input: `[N, C_in, H, W]` (batch × channels × height × width)
- CONV kernel: `[C_out, C_in, KH, KW]`
- Output: `[N, C_out, H_out, W_out]`

2D inputs continue to work — the ops detect dimensionality and dispatch accordingly. This makes NML programs generated by the ONNX transpiler (Phase 4) directly runnable without shape manipulation.

### Cross-Entropy Loss

`LOSS #2` implements cross-entropy: `L = -Σ y_true * log(y_pred)`. Previously only MSE (`LOSS #0`) was implemented. Required for classification tasks from ONNX-imported models.

### Configurable GPU Thresholds

GPU dispatch crossover points are now configurable at runtime via environment variables. No recompile needed:

| Variable | Controls |
|----------|----------|
| `NML_CUDA_MMUL_THRESHOLD` | Min elements before routing MMUL to CUDA |
| `NML_CUDA_EW_THRESHOLD` | Min elements before routing elementwise ops to CUDA |
| `NML_SYCL_MMUL_THRESHOLD` | Min elements before routing MMUL to SYCL |
| `NML_SYCL_EW_THRESHOLD` | Min elements before routing elementwise ops to SYCL |

**Tests:** `make test-phase1` — `17_broadcast.nml`, `10b_vision_4d.nml`, `15b_loss_ce.nml`, plus regressions on all existing vision and training tests.

## Phase 2 — GPU Coverage

**Goal:** GPU acceleration for the operations that dominate inference cost in real CNNs and transformers — specifically CONV, ATTN, and the normalization/activation ops that follow them.

### CUDA Backend (nml_backend_cuda.cu)

New GPU kernels for NVIDIA hardware:

| Op | Implementation |
|----|---------------|
| CONV | im2col transformation + cuBLAS batched GEMM (handles 4D NCHW from Phase 1) |
| ATTN | Chained GEMM (Q×K^T, scale, softmax, ×V) on GPU |
| GELU | Elementwise GELU approximation kernel |
| SOFTMAX | Numerically stable softmax (max-subtract then exp/sum) |
| LAYERNORM | Single-pass mean + variance + normalize kernel |

### SYCL Backend (nml_backend_sycl.cpp)

Same op set implemented with Intel SYCL (DPC++) for Intel Arc and Iris Xe GPUs. The SYCL backend also gains the new CONV4D path that was missing before Phase 2.

### Elementwise Kernel Fusion

When a CONV or MMUL instruction is immediately followed by RELU, SIGM, TANH, or GELU on the same destination register, the runtime fuses the pair into a single GPU kernel dispatch. This eliminates one round-trip to global GPU memory per fused pair — measurable on large batch sizes.

Fusion is detected at instruction decode time: if the next instruction is an elementwise activation on the same register, the GPU kernel writes through the activation function directly rather than storing and re-loading.

**Tests:** `make test-phase2` — `18_gpu_conv.nml` (CPU path validates correctness), `19_fusion.nml` (fusion code path), plus Phase 1 regression.

## Phase 3 — Training Depth

**Goal:** Make NML's on-device training practical for deeper architectures and production training loops.

### N-Layer TNET

Old `TNET #epochs #lr` trained a fixed 2-layer architecture.
New `TNET R_config #epochs` reads architecture from a config tensor:

```
R_config: shape [L, 3] where each row = [input_size, hidden_size, output_size]
```

Supports 1–8 layers. Each hidden layer uses ReLU activation; the final layer is linear. The config register allows changing network depth at runtime without changing the NML program — useful when the network architecture is itself a data-driven decision.

### BN Opcode — Batch Normalization

| Syntax | Form |
|--------|------|
| Classic | `BN Rdst Rsrc Rgamma Rbeta` |
| Symbolic | `⊞ Rdst Rsrc Rgamma Rbeta` |
| Verbose | `BATCH_NORM dest=Rdst src=Rsrc gamma=Rgamma beta=Rbeta` |

Normalizes `Rsrc` across the batch dimension (first axis): subtract mean, divide by std, then apply learned scale `Rgamma` and shift `Rbeta`.

### DROP Opcode — Inverted Dropout

| Syntax | Form |
|--------|------|
| Classic | `DROP Rdst Rsrc #rate` |
| Symbolic | `≋ Rdst Rsrc #rate` |
| Verbose | `DROPOUT dest=Rdst src=Rsrc rate=#rate` |

Implements inverted dropout: during training, randomly zeroes elements with probability `rate` and scales survivors by `1/(1-rate)`. Pass `#0.0` to disable (inference mode). The inversion ensures outputs have the same expected magnitude regardless of rate.

### WDECAY — L2 Weight Decay (AdamW)

Previously declared but not implemented. Now fully wired into the AdamW optimizer path: `WDECAY #lambda` applies L2 regularization to all weight registers before the Adam update step.

**Opcode count:** 87 → 89 (BN and DROP added).

**Tests:** `make test-phase3` — `20_nlayer_tnet.nml`, `21_batchnorm.nml`, `22_dropout.nml`, `23_wdecay.nml`, plus regression on anomaly detector and training opcode suite.

## Phase 4 — ONNX Import

**Goal:** Allow pre-trained models from any ONNX-compatible framework (PyTorch, TensorFlow, scikit-learn, etc.) to be deployed via NML without manual port.

### nml_from_onnx.py

`transpilers/nml_from_onnx.py` converts an ONNX model to:
1. An NML program (`.nml` file) with register allocation and instruction sequence
2. A weight data file (`.nml.data`) containing all initializer tensors

**Register allocation strategy:**
- R0 = primary model input (always)
- R1–R28 = intermediate activations (allocated in order of first use)
- R29–R31 (RT, RU, RV) = scratch (reserved)
- Named memory slots (`@name`) = weight tensors (initializers)

**Supported ONNX ops (28 total):** Conv, Gemm, MatMul, BatchNormalization, Relu, Sigmoid, Tanh, Gelu, Softmax, MaxPool, AveragePool, GlobalAveragePooling, Add, Sub, Mul, Div, Reshape, Transpose, Flatten, Squeeze, Unsqueeze, Concat, Dropout, LayerNorm, InstanceNorm, PRelu, LeakyRelu, Clip.

```bash
python3 transpilers/nml_from_onnx.py model.onnx
python3 transpilers/nml_from_onnx.py model.onnx --output-nml out.nml --output-data out.nml.data
python3 transpilers/nml_from_onnx.py model.onnx --dry-run
```

### nml_to_hailo.py — CONV4D Fix

The Hailo transpiler previously aborted when it encountered a CONV instruction with 4D tensors. Updated to handle NCHW CONV4D by correctly computing output dimensions and emitting the right HEF configuration.

### test_onnx_import.py

End-to-end test: generates a minimal ONNX model with known weights, transpiles it to NML, runs it with the NML binary, and checks the output against numpy reference values.

**Tests:** `make test-phase4` — runs `tests/test_onnx_import.py`. Skips gracefully if `onnx` is not installed.

## Phase 5 — C API and Embeddability

**Goal:** Enable NML to be embedded as a library in other applications — without spawning a subprocess, without global state, and with a minimal, stable API surface.

### runtime/nml_api.h

Ten public functions. No global state — each `nml_vm_t*` handle is fully independent.

| Function | Purpose |
|----------|---------|
| `nml_vm_create()` | Allocate a new VM instance |
| `nml_vm_destroy(vm)` | Free all resources |
| `nml_vm_load(vm, path, data_path)` | Load program + weights from files |
| `nml_vm_load_mem(vm, prog_text, data_text)` | Load from in-memory strings |
| `nml_vm_set_input(vm, name, data, shape, ndim)` | Write an input tensor |
| `nml_vm_run(vm)` | Execute until HALT (returns 0 on success) |
| `nml_vm_get_output(vm, name)` | Read a named output tensor (view into VM memory) |
| `nml_vm_get_register(vm, index)` | Read a register tensor by index (0–31) |
| `nml_vm_last_error(vm)` | Get the last error string |
| `nml_version()` | Runtime version string |

`NMLTensor` is the return type for output/register access: `{ const float *data, int ndim, int shape[4], int size }`. The data pointer is valid until `nml_vm_destroy()` or the next `nml_vm_run()` — copy if you need it beyond that.

### Shared Library Builds

| Target | Output | Use case |
|--------|--------|----------|
| `make libnml.so` | `libnml.so` | Linux shared library (ctypes, dlopen) |
| `make libnml.dll` | `libnml.dll` | Windows DLL (ctypes, LoadLibrary) |
| `make libnml.a` | `libnml.a` | Static library (link into binary) |

All three use `-DNML_BUILD_LIB` which disables `main()` and enables the API entry points.

### python/nml.py

Python binding with two operational modes:

**Library mode** (fast, no subprocess overhead): Loads `libnml.so`/`libnml.dll` via ctypes, exposes the `VM` class with `load()`, `set_input()`, `run()`, `get_output()`, `get_register()` methods. Returns numpy arrays when numpy is available, plain Python lists otherwise.

**Subprocess fallback** (always works): `run_program()` and `infer()` spawn the NML binary. Input tensors written to a temporary `.nml.data` file, stdout returned. Automatically selected when the shared library is not found.

`infer()` tries library mode first, falls back to subprocess transparently.

**Tests:** `make test-phase5` — Python import check, subprocess mode run, `tests/test_python_binding.py`.
